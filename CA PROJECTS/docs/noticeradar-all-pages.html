What is NoticeRadar?

A specialized, secure web application for Chartered Accountants that automates the entire lifecycle of government tax notices — from fetching to final response.

The Problem CAs Face

Managing hundreds of clients, each with notices from multiple government portals (IT, GST) is chaotic and time-consuming:

  • Constantly log into different client portals manually
  • Read complex 10-page legal notices
  • Chase clients over email for required documents
  • Draft legally formatted replies under strict deadlines
  • Submit responses and track resolution status
  • Risk missing deadlines → penalties for clients
🎯

The NoticeRadar Solution

NoticeRadar automates the entire compliance lifecycle end-to-end:

  • Automatically fetches notices from government portals (RPA)
  • AI summarizes complex legal documents in seconds
  • Secure client portal for document uploads
  • AI drafts legally formatted response letters
  • Interactive calendar tracks all deadlines visually
  • Full forensic audit trail for compliance evidence

02 — Architecture

System Architecture

How all the different pieces of NoticeRadar connect — from external government portals to user interfaces.

Fig 1.0 — High-Level System Architecture Block Diagram

The Four System Zones

🏛️ External World

The Government Income Tax Department portal and GST portal. These are third-party systems that NoticeRadar connects to via automated RPA scripts — logging in, checking for new notices, and downloading PDFs. NoticeRadar never controls these portals, it only reads from them.

🖥️ NoticeRadar Backend

The core brain running on Node.js + Express. Handles all business logic, enforces security (JWT auth, rate limiting, CORS), talks to MongoDB, calls AI APIs, and manages file storage. Every request from both users passes through this server.

🗄️ Data & Storage Layer

MongoDB stores all structured data (users, notices, chats, audit logs) as flexible JSON documents. AWS S3/MinIO stores all actual files (PDFs, images). These two systems together form the persistent memory of NoticeRadar.

👥 Users

Two completely separate frontend interfaces. The CA Dashboard (full-featured React app) for staff and admins. The Client Portal (simplified interface) for Tax Clients. Both communicate with the same backend server via authenticated HTTPS API calls.

Data Flows — What Goes Where

🔁

Gov Portal → Server (Async, Nightly)

The RPA automation script runs on a schedule. It uses stored, encrypted credentials to log into government portals, scrapes the notice list, downloads new PDFs, extracts metadata (Notice #, PAN, Assessment Year, Due Date), and saves to MongoDB. This happens without any human intervention.

↔️

Server ↔ MongoDB (Real-time, Bidirectional)

Every CA action — viewing a notice, creating an action request, adding a comment — triggers a read or write to MongoDB via Mongoose. The data is structured according to strict schemas (Notice.js, User.js, AuditLog.js) before it is ever saved.

🧠

Server ↔ AI Engine (On-demand)

When a CA opens the Notice Drawer, the server sends the raw notice text to the AI API. When the CA opens Response Studio, the server sends notice context + documents. The AI returns summaries or draft letters. These calls are made on-demand, not continuously.

🗂️

Server ↔ AWS S3 (On file events)

When a client uploads a document, Multer processes the file on the server, which then streams it to S3 with server-side encryption. When a CA needs to view a file, the server generates a time-limited pre-signed URL from S3. Files are never publicly accessible.

Frontend — What You See in the Browser

The frontend is a React single-page application (SPA). When you navigate between pages, the browser doesn't reload — React swaps out components instantly, making the app feel native and fast.

React.jsUI Framework

The foundation. Builds fast, interactive user interfaces using reusable "components." When data changes, only the affected part of the screen re-renders — not the entire page. This is what makes NoticeRadar feel instant.

React RouterNavigation

Controls navigation without page reloads. When you click "Notices," the URL changes to /notices and React Router swaps the component — the user feels like they moved to a new page but the browser never refreshed.

Tailwind CSSStyling

A utility-first CSS framework. Instead of writing custom CSS for every element, you apply small pre-built classes directly in HTML (e.g., "flex", "rounded-lg", "text-blue-500"). Results in consistent, responsive designs.

AxiosAPI Client

The HTTP client that sends requests from the browser to the backend server. Automatically attaches the user's JWT Bearer token to every request header so the server knows who is making each call.

TanStack QueryState/Cache

Manages server data caching intelligently. If you opened the Notice Inbox 30 seconds ago, it serves the cached result instantly instead of making another API call. Knows when to auto-refresh stale data.

RechartsCharts

Renders the Dashboard analytics as interactive charts. Powers the "Notices Resolved vs Pending" bar charts, overdue trends, and average response time metrics seen by admins.

Lucide ReactIcons

A library of clean, consistent SVG icons used throughout the interface. Every icon (search magnifier, upload arrow, user profile, calendar) comes from this single source, ensuring visual consistency.

React Hot ToastNotifications

The small pop-up notifications that appear in the corner when actions succeed or fail. "Notice marked as resolved ✓" or "Upload failed — file too large." Disappear automatically after a few seconds.

Backend — The Server Brain

The backend is a Node.js + Express REST API. It processes all requests, enforces security, talks to the database and AI, and sends structured JSON responses back to the frontend.

Node.jsRuntime

The JavaScript engine that runs on the server. Allows using the same language (JavaScript) on both front and backend. Excellent for handling many simultaneous I/O operations (multiple users uploading files, querying the database, calling APIs).

Express.jsFramework

The web framework that sits on top of Node. It listens for incoming HTTP requests (GET /api/notices), routes them to the right handler function, runs middleware security checks, and sends responses back to the client.

MongooseODM

Object Data Modeling library. Acts as a translator between Node.js objects and MongoDB documents. Lets you define strict schemas (e.g., "PAN must be exactly 10 characters") that MongoDB enforces before saving any data.

JWTAuth Tokens

JSON Web Tokens are the digital "ID cards." When you log in, the server creates a signed JWT containing your user ID and role. You send this token with every subsequent request. The server verifies the signature to confirm your identity without checking the database each time.

BcryptPassword Hashing

One-way password hashing algorithm. When a user sets a password, bcrypt converts it into a fixed-length hash. Only the hash is stored — never the actual password. Even database admins cannot read passwords. Login works by re-hashing the input and comparing hashes.

Crypto (AES-256)Encryption

Two-way encryption using the AES-256 standard (same used by banks and governments). Used specifically to encrypt IT portal credentials before storing them. The server can decrypt them when needed for RPA sync, but they are unreadable in the database itself.

ZodValidation

Schema validation library that runs before any data is processed. Ensures every API request contains valid data: email fields are actually email-formatted, required fields exist, numbers are numbers. Also validates that all environment variables (API keys, secrets) are present at server startup.

MulterFile Upload

Middleware that handles multipart/form-data — the format browsers use to upload files. Configures strict upload rules: max file size of 10MB, only PDF or image file types accepted. Malicious files (scripts, executables) are rejected before they reach the server logic.

Database & Storage

MongoDB NoSQL Database

Stores all structured application data as flexible JSON-like "documents." Unlike rigid spreadsheet-style SQL databases, MongoDB can store varying shapes of data — perfect for notices that might have different fields for IT vs GST types.

Stores: User profiles, Notice records, Client data, Chat messages, Action requests, Audit log entries, Templates.

AWS S3 / MinIO Object Storage

Object storage used as the secure "Document Vault." MongoDB stores metadata about files (filename, uploader, linked notice), while S3 stores the actual file bytes. Files are encrypted at rest in S3. Access is controlled via short-lived pre-signed URLs.

Stores: Notice PDFs, OCR scan images, Client-uploaded documents, Final reply PDFs on CA letterhead.

Security Libraries

Five dedicated security libraries are installed and active on every request — they run as middleware, meaning they intercept and check every API call before the main handler ever runs.

HelmetSecurity

Sets 11 HTTP security headers on every response. Prevents browsers from being tricked by attackers. Hides that the server uses Node.js (X-Powered-By header removed), prevents MIME sniffing, blocks clickjacking, and enforces HTTPS upgrades.

mongo-sanitizeInjection Defense

Strips MongoDB operators from request bodies. Prevents attackers from injecting database commands (like {"$gt":""}) into form fields to bypass login checks or extract all records from the database.

xss-cleanXSS Defense

Sanitizes all incoming text data to remove malicious JavaScript. Prevents attackers from typing <script> tags into comment boxes that would then run in other users' browsers when the comment is displayed.

rate-limitBrute Force

Counts login attempts per IP address. After exceeding the threshold (e.g., 10 failed attempts in 15 minutes), that IP is locked out for a cooldown period. Stops automated tools from rapidly guessing thousands of passwords per second.

corsDomain Control

Cross-Origin Resource Sharing policy. Configures the server to accept API requests only from the officially configured frontend domain. Any request from an unknown domain receives an instant rejection — no data is returned, no processing happens.

🤖 Zone 1 — Automation & AI

These 4 features eliminate manual repetitive tasks and bring AI intelligence to every notice interaction.

#01

Automated Portal Sync (RPA Automation)

Securely stores each client's Income Tax Portal login credentials (AES-256 encrypted). Every night, the server acts like a robotic user — silently logs into incometax.gov.in, checks the notice list, downloads any new PDFs, extracts metadata (Notice #, PAN, Assessment Year, Due Date), and saves everything to the database. Zero manual effort from the CA.

#02

Quick Scan & Auto-Extract (OCR)

A client hands a physical paper notice to the CA. The CA scans it to PDF or image and uploads it via "Quick Scan" in NoticeRadar. The OCR (Optical Character Recognition) engine reads the document pixel-by-pixel, converts the image into structured text, and automatically populates all notice form fields — Notice Number, PAN, Assessment Year, Due Date. No manual data entry required.

#03

AI Notice Summarization

When a CA opens the Notice Drawer, NoticeRadar instantly sends the full raw notice text to the AI engine (powered by Claude/GPT). The AI returns a 3-bullet summary in plain language: the applicable legal section (e.g., "Section 143(1) Intimation"), the specific issue (e.g., "TDS mismatch — department found discrepancy vs Form 16 filed"), and the exact deadline. A CA understands the entire notice in 10 seconds without reading 10 pages of legal text.

#04

AI Response Studio

When the CA clicks "Auto-Draft" in the Response Studio, NoticeRadar bundles the original notice summary together with all uploaded client documents and sends them to the AI. The AI generates a complete, professionally formatted legal reply letter: proper salutation, subject line referencing the notice number and assessment year, detailed body paragraphs, references to the attached documents, and a formal closing. The CA reviews it in a rich text editor, makes any corrections, and approves.

📋 Zone 2 — Workflow Management

Tools to organize, prioritize, and act on notices across hundreds of clients efficiently.

#05

Centralized Notice Inbox

A single interface showing all notices from all clients in one place. Color-coded urgency indicators (🔴 Red = Due Tomorrow, 🟡 Yellow = Due This Week, 🟢 Green = More than a week). Sortable by status (Pending, In Progress, Resolved), department (Income Tax vs GST), assigned staff member, client name, and PAN. The primary command center for CA staff — replaces having to log into dozens of different portals.

#06

Notice Drawer

A full-featured slide-out side panel that opens without leaving the inbox when you click on any notice. Contains everything in one view: all metadata (PAN, Assessment Year, Section, Due Date), the AI-generated summary, all document attachments (both downloaded notices and uploaded proofs), a complete activity timeline (every action logged with timestamp), and the full communication thread with the client. No context-switching, no separate screens.

#07

Interactive Notice Calendar

A full-page calendar view (monthly/weekly layout) that plots every notice on its exact government-issued due date. At a glance, see which days have heavy deadline loads, identify overdue notices in red, and plan the workload for the entire firm for the month ahead. Clicking a date shows all notices due that day. Prevents the common problem of discovering a deadline has passed only when it's too late.

#08

Bulk Actions

Select multiple notices simultaneously in the inbox using checkboxes. Then assign all selected notices to a specific staff member in one click — or update the status of all selected notices at once. Essential when a new staff member joins the firm and you need to distribute a workload, or when completing end-of-month status updates across dozens of notices.

💬 Zone 3 — Client Collaboration

#09

Action Requests

CAs create formal, tracked document requests tied directly to a specific notice. Example: "Please upload your rent receipts for April–March 2023, Form 16 from employer, and bank statements for the period." The request is logged in the system with a status (Pending / Fulfilled), and the client receives an automated email with a direct link to upload. No chasing on WhatsApp or email. Full accountability trail.

#10

Dedicated Client Portal

A completely separate, simplified login at a different URL — designed for Tax Clients, not CAs. Clients see only their own notices, their current status ("Your CA is reviewing this"), and any pending document requests in plain language ("Please upload: Form 16 PDF"). They can upload files directly, track progress, and communicate — without ever seeing the complex CA workflow. Can be onboarded in 2 minutes via email invite.

#11

Comments & Real-time Chat

A threaded conversation lives inside each Notice Drawer. Both CA staff and the specific client can post messages, ask questions, clarify details, and share updates — all in context of the specific notice. Every message is timestamped and permanently stored. Replaces the chaos of long email chains where the attachment and context get lost after 20 replies. All communication is searchable and audit-ready.

#12

Automated Email Alerts

The system sends automated email notifications for every significant event — no manual sending required. Triggers include: notice approaching deadline (3-day warning, 1-day warning), new document uploaded by a client, Action Request fulfilled, notice status changed, and notice marked as Resolved. Both CAs and clients receive relevant alerts, keeping everyone informed without creating extra work.

⚙️ Zone 4 — Administration

#13

Dashboard & Analytics

High-level firm performance overview rendered as interactive Recharts visualizations. Metrics shown: Total Notices by Status (Pending / In Progress / Resolved), Overdue count with trend, Average response time per staff member, Notices split by department (Income Tax vs GST), Resolution rate for the month, and most active clients. The admin can assess firm health at a glance without manually counting records.

#14

User & Role Management

Super Admins can invite new staff members via email, assign their role (Super Admin vs Staff), set which clients they manage, and revoke access instantly when someone leaves the firm. Role-Based Access Control is enforced at the API level — not just the UI. A Staff member cannot see another client's notices even by calling the API directly. Access control is structural, not cosmetic.

#15

Client Directory Management

A dedicated page to manage the CA firm's entire client base. Each client record stores: Full legal name, PAN card number, Entity type (Individual, Company, LLP, Partnership, HUF), Contact email, IT portal login credentials (stored AES-256 encrypted), linked staff member, and all associated notices. CAs can add new clients, edit details, and remove clients — keeping the firm's database accurate and up to date.

#16

Template Management

Admins can create, name, and save standardized boilerplate reply templates for common notice types. Examples: "Standard GST Reply Template", "Section 143(1) TDS Mismatch Template", "HRA Exemption Proof Submission Template." When CA staff opens Response Studio, they can load any saved template as the starting point, then customize and fill in the client-specific details. Saves significant time on routine notices.

🛡️ Zone 5 — Security & Auditing

#17

Document Vault (AWS S3)

All uploaded documents — Notice PDFs, scanned images, client-submitted proofs (Form 16, rent receipts, bank statements), and final reply PDFs — are stored in a dedicated AWS S3 bucket with server-side encryption. Files are never publicly accessible. When a user needs to view a file, the server generates a time-limited pre-signed URL (expires after 15 minutes) that works for that specific file. After expiry, the URL is dead — it cannot be reused or shared.

#18

Forensic Audit Trail

Every single state-changing action in the system is permanently, immutably logged in the AuditLog collection with full context: the user's ID and name, their IP address, the exact timestamp, the action performed, and what data changed. Logged events include: user logins and logouts, notice creation and deletion, document uploads and downloads, portal syncs, status changes, and any admin actions. Super Admins can view the full forensic log — essential for regulatory compliance and internal accountability.

#19

Multi-Factor Authentication (MFA)

Users can link Google Authenticator (or any TOTP app) to their account. Once enabled, login requires two factors: (1) the correct password, AND (2) the current 6-digit TOTP code from the authenticator app — which rotates every 30 seconds. Even if a password is leaked via phishing or a data breach elsewhere, the account cannot be accessed without the physical phone that has the authenticator app. Optional but strongly recommended for all users handling sensitive financial data.

All 8 Security Layers

Each layer operates independently, defending against a specific class of attack. All 8 are active simultaneously on every production request.

🔐

Role-Based Access Control

auth.middleware.js

Every API route is protected by auth.middleware.js which verifies the JWT token AND checks if the user's role has permission to access that specific resource. A Staff member attempting to fetch another client's notices receives a 403 Forbidden response — even if they have a valid JWT token.

🔒

AES-256 + Bcrypt Encryption

crypto · bcrypt

Bcrypt (one-way): passwords are transformed into irreversible hashes. Even Anthropic cannot read stored passwords — login works by re-hashing the input and comparing hashes. AES-256 (two-way): IT portal credentials are encrypted before storage. The server holds the decryption key in an environment variable — separate from the database. Both the database and the key must be compromised simultaneously for credentials to be exposed.

📱

Multi-Factor Authentication

TOTP / RFC 6238

Google Authenticator uses the TOTP standard (Time-based One-Time Password, RFC 6238). A 6-digit code is derived from a shared secret + the current Unix timestamp, rotated every 30 seconds. Even with a stolen password, an attacker cannot log in without the physical device running the authenticator app generating the current code.

🛡️

Helmet HTTP Security Headers

helmet

Sets 11 HTTP headers on every response: removes X-Powered-By (hides Node.js), sets Content-Security-Policy (prevents XSS), X-Frame-Options: DENY (blocks clickjacking), X-Content-Type-Options (prevents MIME sniffing), Strict-Transport-Security (enforces HTTPS), Referrer-Policy, and more. These prevent entire classes of browser-based attacks at the HTTP protocol level.

💉

NoSQL Injection Prevention

express-mongo-sanitize

In MongoDB, attackers can type operators like {"$gt":""} into form fields to bypass password checks (the query becomes "find user where password is greater than empty string" — which returns every user). mongo-sanitize removes all keys starting with $ from req.body, req.query, and req.params before the request reaches any controller.

🚫

XSS Cross-Site Scripting Protection

xss-clean

Stored XSS attack: an attacker types <script>document.cookie</script> into a comment field. If stored and displayed to other users without sanitization, it executes in their browser, stealing their session cookies. xss-clean strips all HTML tags and script content from incoming text before any data is stored in MongoDB.

⏱️

Rate Limiting

express-rate-limit

Configures a sliding window rate limiter on the /api/auth/login endpoint. Example configuration: 10 failed attempts per 15 minutes per IP address. After the threshold is exceeded, the IP receives a 429 Too Many Requests response for the remainder of the window. Automated password-guessing tools (which can try thousands of combinations per second) are defeated — they're locked out after just 10 attempts.

🌐

CORS Domain Allowlisting

cors

Cross-Origin Resource Sharing policy: the server explicitly allows only the configured frontend domain in its CORS whitelist (e.g., https://app.noticeradar.com). Any request from a different origin — a malicious website, a hacker's script, a rogue third-party app — receives a CORS error in the preflight check. The actual request is never processed. No data is ever returned to unauthorized origins.

07 — Structure

Project Structure

Every folder and file in the codebase — what it does, why it exists, and how it connects to the rest of the system.

Fig — Full Project Folder Architecture

Server/ — Every File Explained

Entry Points

src/app.js

Creates and configures the Express application. Registers all global middleware in order: Helmet (security headers) → CORS (domain allowlist) → express-rate-limit → express-mongo-sanitize → xss-clean → JSON body parser → routes. The order matters — security guards must run before any route handler.

src/server.js

The actual startup file. Calls db.js to connect to MongoDB, then starts the Express server listening on the configured port. Also handles graceful shutdown — if the process is killed, it closes the MongoDB connection cleanly before exiting.

src/config/

config/db.js

Contains the Mongoose connection logic. Reads the MongoDB connection string from environment variables (never hardcoded) and establishes the database connection. Includes retry logic and connection event handlers for production resilience.

config/env.js

Uses Zod to define and validate all required environment variables at startup: MongoDB URI, JWT secret, AWS credentials, AI API key, encryption key, frontend URL. If any variable is missing or malformed, the server refuses to start and logs exactly which variable is wrong. Prevents silent failures in production.

src/models/

models/Notice.js

MongoDB schema for a notice. Required fields: PAN (10 chars), Assessment Year, Department (IT/GST), Notice Number, Due Date, Status (enum), PDF URL (S3 link), linked client ID, assigned staff ID. Timestamps auto-added. Indexes on status + dueDate for fast inbox sorting.

models/User.js

Schema for CA staff and admin accounts. Fields: name, email (unique), hashed password, role (enum: super_admin / staff), MFA secret (if enabled), linked client IDs, createdAt. The password field has select:false — it's excluded from all queries by default, requiring explicit .select('+password') to read it.

models/AuditLog.js

Schema for the forensic audit trail. Immutable records (no update/delete allowed). Fields: userId, userName, action (enum of all possible actions), entityType, entityId, ipAddress, userAgent, timestamp. TTL index not set — logs are permanent and never auto-deleted.

models/Client.js

Schema for CA's tax clients. Fields: fullName, PAN, entityType (enum), email, phone, portalCredentials (AES-256 encrypted JSON), assignedStaffId, createdAt. The portalCredentials field is always encrypted before save and decrypted on read — handled transparently in the service layer.

src/controllers/ & src/routes/

routes/ → controllers/ pattern

Routes are thin: they map URL + method to a controller function. Example: GET /api/notices → noticesController.getAll. Controllers are also thin: they extract data from req, call a Service function for business logic, and send the formatted response. All heavy logic lives in Services — this makes testing and maintenance clean.

src/services/

services/ai.service.js

Wraps calls to the AI API (Claude/GPT). Two main functions: summarizeNotice(noticeText) → returns 3-point summary, and generateReply(noticeContext, documents) → returns draft letter. Handles API retries, timeout errors, and rate limit responses from the AI provider.

services/auth.service.js

Handles all authentication logic: validateCredentials (bcrypt.compare), createToken (JWT sign), verifyToken (JWT verify), validateMFACode (TOTP verification). Keeps auth logic out of controllers — controllers just call these functions and trust the result.

services/encryption.service.js

Two functions: encrypt(plaintext) → returns AES-256-GCM ciphertext + IV + authTag as a combined string, and decrypt(encryptedString) → returns original plaintext. Used exclusively for IT portal credentials. The encryption key is read from environment variables — never hardcoded in source code.

web/ — Every File Explained

Core Files

src/index.js

The browser entry point. Renders the root React component into the DOM's #root div. Also wraps the app in TanStack QueryClientProvider (for data caching) and BrowserRouter (for React Router navigation).

src/App.js

The router configuration. Defines all URL routes: / → Dashboard, /notices → NoticeInbox, /calendar → NoticeCalendar, /clients → ClientList, /admin → AdminPanel, /client-portal → ClientPortal, /login → Login. Wraps most routes in <ProtectedRoute> to enforce authentication.

src/api/axios.js + endpoints.js

axios.js creates the Axios instance with the backend base URL and adds a request interceptor that automatically reads the JWT from localStorage and adds it as Authorization: Bearer [token] to every outbound request. endpoints.js is a constants file mapping human-readable names to API URL strings — avoids hardcoding URLs throughout the codebase.

src/auth/

auth/AuthContext.js

React Context that holds the global authentication state: currentUser object, isAuthenticated boolean, login() function (stores JWT, sets user), and logout() function (clears JWT, resets state). Any component in the app can consume this context to check who is logged in without prop-drilling.

auth/ProtectedRoute.js

A React wrapper component. Before rendering the child component, it checks AuthContext.isAuthenticated. If false, it immediately redirects the browser to /login using React Router's Navigate. Used to wrap every route that requires a logged-in user — one line of protection per route.

src/pages/

pages/Admin/ — SuperAdmin Only

AuditLogs.js / ForensicLogs.js: Fetches and displays the full AuditLog collection in a searchable, filterable table. Shows: user name, action, IP address, entity affected, timestamp. Super Admin only — all other roles are redirected by ProtectedRoute. UserManagement.js: Lists all staff accounts with ability to invite new users via email, change roles, and revoke access.

pages/Notices/ — The Core Feature

NoticeInbox.js: Main notice list with filters and sort. NoticeDrawer.js: Slide-out panel with full notice details, AI summary, documents, activity log, and chat thread. NoticeCalendar.js: Full-page calendar using a calendar library, notices plotted by due date. QuickScanNotice.js: File upload page with OCR processing flow and auto-populated form fields.

pages/Responses/ResponseStudio.js

The AI drafting interface. Shows the notice summary on the left, attached documents in the middle, and the rich text editor for the AI-generated reply on the right. "Auto-Draft" button triggers the AI API call. Rich text editor supports bold, italic, tables, and lists. "Approve" generates the final PDF.

pages/ClientPortal/

ClientDashboard.js: Simplified dashboard showing the client's notices in plain language with status indicators. No CA-specific features visible. ClientNoticeDetails.js: Individual notice page for the client — shows the pending Action Request, the upload interface, and the chat thread with their CA.

pages/Tools/PortalSync.js

The RPA trigger page. Allows CA admin to manually trigger a portal sync for specific clients on-demand (instead of waiting for the nightly schedule). Shows sync history: last sync time, number of new notices found, any errors encountered during the RPA process.
